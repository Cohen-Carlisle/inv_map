## Architecture

`InvMap` (single module, `lib/inv_map.ex`) is a bidirectional map. The struct holds two fields kept in sync:

```elixir
defstruct forward: %{}, inverse: %{}
```

`forward` is the user-facing `key => value` map; `inverse` mirrors it as `value => key`. Lookups (`fetch/2`, `fetch!/2`, `get/3`, `has_key?/2`, `delete/2`) are **symmetric** ‚Äî they pattern-match against both maps, so `get(m, key)` and `get(m, value)` both work and return the counterpart.

### The involution-on-get invariant

The defining property of `InvMap`: lookup is involutive. If `f(x) = get(m, x)`, then `f(f(x)) === x` for every key ‚Äî exactly the check in `validate_involution_on_get!/1`. Equivalently, no two keys map to the same value, and a key cannot collide with a value.

The invariant is enforced in two different ways depending on the entry point:

- **Constructors (`new/1`, `new/2`)** *validate* via `validate_involution_on_get!/1` (private) and raise `ArgumentError` if the input violates it. Examples: `InvMap.new(%{1 => 2, 2 => 3})` raises (`f(f(1)) == 3`, not `1`); `InvMap.new(%{a: 1, b: 1})` raises (two keys ‚Üí same value).
- **`put/3`** *repairs* by deleting any existing entries that would conflict with the new pair before inserting it. So `put(InvMap.new(%{1 => 1.0, 2 => 2.0}), 1, 2.0)` returns `InvMap.new(%{1 => 2.0})` ‚Äî the prior `2 => 2.0` entry is dropped to keep the invariant.

This split is intentional. Don't move validation into `put/3` or repair into `new/*`.

Additionally, both entry points reject inverse pairs: `{k, v}` and `{v, k}` are redundant because the inverse map already encodes the reverse direction. `new/*` deduplicates them via a private `dedup_inverse_pairs/1` (which side survives is undefined); `put/3` collapses them implicitly by deleting both `key` and `value` from the existing maps before inserting. Constructed InvMaps therefore never have `{a, b}` and `{b, a}` simultaneously in their internal maps.

### Custom `Inspect`

`defimpl Inspect` renders an `InvMap` as `InvMap.new(%{...})` (showing only the forward map). This shape is what the doctests assert against, so changing it breaks doctests.

## Conventions / gotchas

- **Doctests document the public API; the test file is comprehensive.** Doctest `## Examples` blocks show consumers the most important behaviors and stay accurate by running via `doctest InvMap`. The test file mirrors those happy-path examples and adds edges, regressions, and behaviors not worth surfacing in user docs. The redundancy is intentional ‚Äî it gives the developer a single comprehensive coverage view.
- **Test names describe what the function does, not what the test checks.** Common shapes: `returns X when Y`, `<verb>s X` (using the function's primary verb ‚Äî `creates`, `puts`, `deletes`, etc.), `raises Y when Z`, or a property statement like `the last key in wins`. Avoid abstract test-intent verbs (`compares`, `distinguishes`, `validates`, `ensures`) ‚Äî those name the test's job, not the function's behavior.
- **`InvMap` is not yet `Enumerable`.** `new/1` and `new/2` special-case `%InvMap{}` inputs (see the head clauses); generic enumerable protocols won't work on it. There are `# TODO: make InvMap enumerable` notes in the test file.
- **Outstanding TODOs in source:**
  - `fetch!/2` raise path lacks a non-doctest test.
  - The `Inspect` impl has a commented-out 1.19+-safer version using `Inspect.Algebra.to_doc_with_opts` that should eventually replace the current `to_doc` call.
- **Beware identity pairs.** Consider InvMaps containing entries like `%{1 => 1, 2 => 2}` ‚Äî the value `1` plays both forward-key and inverse-key roles, which is easy to miss when implementing. If the identity pair case is a credible source of regression, add a regression test for it.
- **Regression tests go at the fix site.** Most public functions in this codebase compose over a smaller core. When a bug is fixed by changing function X, add one test on X ‚Äî the smallest function that fails without the fix. Don't mirror the test across every caller; here, that's redundant coverage, not extra signal.
